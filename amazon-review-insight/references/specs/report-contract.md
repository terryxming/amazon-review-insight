# Report Contract

report_contract_version: v0.3.2

## Product Design

HTML 报告视觉样式必须使用 Product Design 插件设计，产物沉淀为：

- `assets/report/design-tokens.json`
- `assets/report/report.css`
- `assets/report/report-layout.md`

`render_report.ts` 只能使用这些资产渲染报告。

HTML 报告允许引入白名单展示型 CDN 来加载前端图标、字体或图表组件，但不得把 Review 数据、Sorftime key、运行时 token 或环境变量值发送到外部服务。未实际使用 CDN 时，报告仍应保持可离线阅读。

## 版本生命周期

报告契约遵循 `references/specs/version-lifecycle.md`。已发布版本必须冻结；任何报告布局、交互、字段、样式、证据高亮或验收口径调整，都必须进入新的 skill 版本，不能回写已经发布的 Git tag。当前报告契约属于已发布的 `v0.3.2`，`v0.3.2` 及更早已发布版本均已冻结。

## 主报告页章节

报告总标题必须独立于章节结构渲染，标题文本为 `{ASIN} Review VOC 决策报告`。该标题不得替代任何一级章节标题。

1. 数据范围与口径。
2. ASIN 元数据。
3. Review 健康度。
4. 关键结论。
5. VOC 主题地图。
6. 关键结论详情页。
7. VOC 主题详情页。
8. 详情页内类型/观点筛选。
9. 机会矩阵与业务动作清单。
10. 限制说明与 checkpoint 状态。

左侧一级导航必须带序号。带有二级导航的一级导航必须使用默认展开的折叠结构，并支持展开/收起；二级导航也必须带层级序号，例如 `4.1 人群`、`5.1 正向主题`。

主报告一级章节标题下方不得渲染说明文字。必要解释必须放入标题问号 tooltip、具体卡片字段或表格字段中。

## 交付文件命名

交付文件必须使用 `YYYYMMDD-ASIN-suffix` 格式，其中 `YYYYMMDD` 为交付日期，`ASIN` 为实际 ASIN。HTML 报告文件名必须为 `YYYYMMDD-ASIN-review-insight-report.html`，Review 编码层 Excel 文件名必须为 `YYYYMMDD-ASIN-review-coding.xlsx`。例如 `20260616-B0CR1R7FKP-review-insight-report.html`。

## 数据范围与 ASIN 元数据

“数据范围与口径”只展示：站点、ASIN、数据来源、抓取时间、已知缺失字段。不得继续在该模块展示 Review 样本数、ASIN 总评论数量、产品星级或分析口径。

“数据范围与口径”模块的正文章节标题必须显示为“数据范围与口径”。不得把 `{ASIN} Review VOC 决策报告` 放在该模块标题位置；它只能作为独立报告总标题展示。

“ASIN 元数据”必须展示在“Review 健康度”之前。字段来自 `product_metadata`，包含 ASIN、品牌、标题、主图、价格、星级、ASIN 总评论数、分类、所属大类、所属细分类目、上架时间。主图必须以图片方式展示；缺失字段显示 `unknown`，不得推断。

## Review 健康度

“Review 健康度”只展示：样本平均星级、4-5 星占比、1-3 星占比、样本评论最早日期、样本评论最新日期、样本各星级分布。星级分布标题必须写为“样本各星级分布”。不得继续展示正文存在率或日期存在率。

## 关键结论

必须覆盖八类洞察：人群、场景、用户任务、购买理由、用户期望、实际体验、满意点、不满意点。缺失时写 `unknown`，不得编造。

每个关键结论卡片必须包含：

- 一句话总结，用于快速理解该维度。
- 类型分布表，用于穷举该维度下的主要、次要、新兴和长尾类型。
- 提及数量和占比，分母为 Review 样本数。
- 判断依据和代表性 evidence。
- 对产品改进、Listing 优化或图片/视频优化的业务含义。

类型分布表的边界：它回答“这个洞察维度内部由哪些类型构成”，不替代 VOC 主题地图。VOC 主题地图回答“哪些跨维度问题或机会需要被业务处理”。

左侧导航必须在“4. 关键结论”下展示八类洞察二级导航，并链接到对应关键结论块。正文中不应额外渲染“八类横向洞察”副标题。关键结论标题右上角必须展示问号 tooltip，用于解释关键结论是跨主题的横向洞察，不是具体问题清单。每个关键结论维度必须是一行一个默认展开的折叠块；折叠后仍保留维度标题、提及数量和核心摘要。HTML 类型分布表不得展示“角色”列或角色 badge；角色字段可继续保留在 Excel 中用于复核。摘要 chip 必须带中文标签，例如“提及：”“置信度：”。

关键结论卡片必须整体可点击，点击卡片任意非链接位置时，当前主报告页不得滚动跳转；浏览器必须在新标签页打开同一个 HTML 内的关键结论详情视图。关键结论详情页必须进入独立路由模式：隐藏目录、主报告其它章节、其它关键结论详情和 VOC 主题详情，仅展示当前关键结论详情。详情页顶部 sticky 面板必须展示所属关键结论、提及数量、置信度、核心结论、业务含义和类型筛选入口。默认展示支撑该关键结论的全部相关 Review 原文、完整中文翻译和句子级证据高亮；点击某个类型后，在当前详情页筛选该类型相关的全量 Review。

## VOC 主题详情页

同一个自包含 HTML 内必须提供主题详情视图。点击 VOC 主题地图中的主题卡片后，浏览器必须在新标签页打开对应主题详情页，原主报告页不得滚动或跳转。每个主题详情页展示：

- 顶部 sticky 详情面板，其中 VOC 主题卡片和观点筛选入口合并在同一个卡片内。
- 完整 Review 原文。
- 完整中文翻译。
- 原文中的完整证据句，并按证据性质使用语义颜色高亮。
- 中文翻译中的对应完整证据句，并使用同一证据性质的语义颜色高亮。

主题详情页默认展示支撑该主题的全部相关 Review。点击某个观点后，不再打开新标签页，而是在当前主题详情页内筛选出与该观点相关的全部 Review。点击“全部主题评论”后恢复主题全量评论。

当 URL hash 为 `#theme-detail-{theme_id}` 时，HTML 必须进入主题详情独立路由模式：隐藏目录、主报告其它章节和其它主题详情 section，仅展示当前主题详情 section。用户在新标签页内向上或向下滚动时，不得看到其它 VOC 主题卡片、主报告章节或无关内容。

主题级信息只在顶部 sticky 详情面板的主题区中完整呈现。观点筛选入口必须与该主题区合并在同一个 sticky 卡片内，使用内部间隔或分割线区分上下内容；不得在 sticky 卡片下方另起一个卡片重复渲染同一主题标题、核心问题、归因假设或业务含义。默认状态只展示筛选控件和当前筛选摘要，点击观点后只展示该观点的统计、判断依据和业务含义。

sticky 详情面板的主题区必须展示主题名称、主题类型、运营优先级、严重度、提及数量和占比，并以明确标签展示 `核心问题`、`归因假设`、`业务含义` 和 `运营动作`。这些主题说明行是同级正文，颜色必须保持一致，不得将其中一行降级为灰色 subtle 文案。合并布局不得以减少重复为由丢弃 `root_cause_hypothesis`。

## VOC 主题地图和主题详情筛选

VOC 主题地图必须按三组展示并穷举全部前台 VOC 主题：

- 正向主题：用户已经认可、可以被运营放大的购买驱动、价值信任或人群适配主题。
- 负向主题：已经造成低星、故障、可靠性或强烈不满的主题，优先用于止损和闭环。
- 未满足的机会点：用户有明确预期但体验、说明、配件、场景或边界还未被充分满足的可补齐机会。

每个前台 VOC 主题必须且只能归入一个主分组。三组主题数量之和必须等于 `voc_themes.length`，不得因为分组而遗漏主题或只展示 Top N。若一个主题同时包含正负信号，agent 必须拆分主题，或按主导业务含义归入最合适的主分组并在主题说明中保留混合证据。

VOC 主题地图必须展示每个主题下的观点分布。每个主题卡片必须包含：

- 主题名称、主题类型、运营优先级、严重度、提及数量和占比。
- 主题核心问题和业务含义。
- 运营动作，用一句话回答运营下一步应该放大、澄清、止损还是排期补齐。
- 主题内观点分布表。
- 主题卡片整体可点击，并在新标签页打开该主题详情页。
- 主报告页中的观点分布表用于预览主题下观点构成；观点筛选入口位于主题详情页内。

左侧导航必须在“5. VOC 主题地图”下展示“5.1 正向主题”“5.2 负向主题”“5.3 未满足的机会点”三类二级导航，并链接到对应分组块。正文中不应额外渲染“主题与观点分布”副标题。VOC 主题地图标题右上角必须展示问号 tooltip，用于解释 VOC 主题地图是业务需要处理的问题/机会集合，并解释运营优先级的含义。每个分组块和每个 VOC 主题必须是一行一个默认展开的折叠块；折叠后仍保留分组标题或主题名称、运营优先级、提及数量和核心问题。主题块内必须展示核心问题、归因假设、业务含义、运营动作、观点分布、3 条完整代表性 Review 和打开主题详情入口。摘要 chip 必须带中文标签，例如“主题类型：”“提及：”“严重度：”“运营优先级：”，不得裸展示 `P0`、`P1` 或 `P2`。

运营优先级不是严重度，也不是给运营看的抽象排序。它表示下一步动作顺序和动作类型：

- `P0`：立即处理。正向购买驱动类主题表示立即放大转化主卖点；负向、风险或预期落差类主题表示立即止损。
- `P1`：本轮迭代处理。通常用于澄清预期、闭环风险或放大已验证机会。
- `P2`：排期优化。通常用于低成本补齐体验、补强信任或进入后续素材/页面/客服优化池。

观点分布表必须包含：

- 观点名称。
- 提及数量和占比，分母为 Review 样本数。
- 极性。
- 判断依据。

HTML 观点分布表不得展示“角色”列；`role` 只作为分析和 Excel 复核字段保留。

点击主题卡片后，当前主报告页不得滚动跳转；浏览器必须在新标签页打开同一个 HTML 内的主题详情视图。主题卡片链接必须使用 `target="_blank"` 与 `rel="noopener"`，并标记新标签打开行为。主题详情页内点击具体观点后，必须在当前标签页筛选出对应观点的全量评论。筛选状态必须包含：

- 顶部 sticky VOC 主题卡片。
- sticky VOC 主题卡片与观点筛选入口必须合并为一个卡片。
- sticky VOC 主题卡片必须展示核心问题、归因假设和业务含义。
- sticky VOC 主题卡片必须展示运营动作。
- 观点名称、提及数量、占比、极性和置信度。
- 观点业务含义。
- 该观点相关的全量 Review 原文。
- 完整中文翻译。
- 原文和译文中的完整证据句高亮。

句子级证据高亮规则：

- 分析 JSON 必须在每条详情 Review 中提供 `evidence_sentences[]`。
- 每条 `evidence_sentences[]` 必须包含 `original`、`translation`、`evidence_type` 和可选 `target`。
- `original` 必须能在该 Review 的完整 `text` 中定位；`translation` 必须能在完整中文译文中定位。
- `evidence_type` 只能是 `positive`、`negative`、`opportunity`、`context`。
- HTML 以语义颜色区分证据性质：正向证据用于卖点和满意点，负向证据用于痛点和风险，机会证据用于未满足需求或预期差，背景证据用于人群、场景、归因或上下文。
- 渲染器必须按当前上下文筛选证据句：关键结论卡片只高亮支撑该关键结论或其类型分布的证据句；VOC 主题卡片只高亮支撑该主题的证据句；观点筛选状态只高亮支撑该观点的证据句。
- 同一条 Review 可以支撑多个结论或主题，但同一次展示不得把其它结论、其它主题或其它观点的证据句带入当前卡片。
- 当旧数据的 Review 级 `evidence_sentences[]` 缺少当前目标时，渲染器必须使用当前关键结论、类型、主题或观点的 `evidence[]` 从完整 Review 中抽取可定位的完整句子作为兜底证据，而不是复用其它目标的高亮句。
- 不得继续生成、保留、导出或渲染 `highlight_terms`、`translation_highlight_terms` 或“原文高亮词/译文高亮词”。

同一 Review 可以同时支撑同一 VOC 主题下的多个观点，因此观点占比允许合计超过 100%。报告必须说明这是多标签提及率。

## 主报告代表性 Review

关键结论卡片和 VOC 主题卡片中的证据区必须展示 3 条不同的完整代表性 Review。每条代表性 Review 必须成对展示：

- 完整英文原文。
- 完整中文译文。

当某个关键结论或 VOC 主题实际可追溯相关评论不足 3 条时，展示全部可追溯评论。不得把关键词、短句、开放标签或 `evidence[]` 片段当作代表性原文。详情页仍展示该关键结论或 VOC 主题相关的全量 Review。

## 机会矩阵与业务动作

机会矩阵与业务动作清单必须按 `action_area` 拆成多个方向块，例如 Listing 优化、产品体验、图片与视频。每个方向块必须独占一行、默认展开、可折叠，并展示动作数量、最高优先级和关联主题摘要。方向块内部使用动作卡片展示业务发现、建议动作、影响指标和验证方式，不得把全部业务动作混在一个横向大表中。

## 电脑端布局

HTML 报告只按电脑端桌面阅读场景设计，不再维护移动端媒体查询或移动端特化样式。主报告最大宽度必须为 1440px，并在 1200px 到 1440px 的电脑端窗口内自然收缩；左侧目录宽度应在桌面端 248px 到 280px 范围内自适应，避免挤压正文。详情页因为隐藏左侧目录，最大宽度使用 1120px，以提升完整 Review 阅读舒适度。主报告应避免一行多个大卡片造成桌面阅读断裂。关键结论、VOC 主题和业务动作方向均采用单列块状布局。数据范围与口径、Review 健康度可以使用桌面端指标网格；Review 健康度应使用星级分布条而不是宽表作为主要展示。长文本、主题 ID、按钮和标签必须在桌面宽度下自然换行；表格列不得窄到形成竖排式阅读。

## 禁止项

- 未经契约允许的外部数据传输型 CDN 或会上传 Review 数据的第三方资源。
- 执行摘要。
- 交付文件未按 `YYYYMMDD-ASIN-suffix` 命名。
- `{ASIN} Review VOC 决策报告` 被渲染为“数据范围与口径”的章节标题，而不是独立报告总标题。
- 数据范围与口径模块没有显示“数据范围与口径”章节标题。
- 数据范围与口径继续展示 Review 样本数、ASIN 总评论数量、产品星级或分析口径。
- ASIN 元数据模块缺失，或没有展示主图、品牌、标题、价格、星级、ASIN 总评论数、分类、所属大类、所属细分类目、上架时间。
- Review 健康度继续展示正文存在率或日期存在率。
- 星级分布标题仍使用“星级分布”而不是“样本各星级分布”。
- 主报告页展示大段完整 Review。
- 完整 Review 编码明细。
- 关键结论只有一句话总结但没有类型分布。
- VOC 主题只有主题卡片但没有观点分布。
- VOC 主题地图没有按正向、负向、未满足的机会点三组穷举全部主题。
- 关键结论卡片无法新标签打开关键结论详情页。
- 关键结论详情页不支持按类型筛选全量相关评论。
- 主报告关键结论或 VOC 主题只展示关键词、短句或 evidence 片段，没有展示完整代表性 Review 原文。
- 主报告关键结论或 VOC 主题代表性 Review 没有展示完整中文译文。
- 关键结论或 VOC 主题地图继续使用一行多个大卡片布局。
- HTML 分布表继续展示“角色”列或角色 badge。
- 左侧一级或二级导航缺少序号。
- 带二级导航的一级导航无法折叠或默认不是展开状态。
- 主报告一级章节标题下方渲染说明文字。
- 关键结论或 VOC 主题地图标题缺少问号 tooltip。
- 摘要 chip 只展示裸值，缺少中文标签说明。
- VOC 主题只裸展示 `优先级：P0/P1/P2`，但没有说明运营动作语义。
- VOC 主题缺少“运营动作”字段，导致运营不知道下一步要放大、澄清、止损还是排期补齐。
- 移动端媒体查询或移动端特化样式。
- 主报告仍使用 1180px、1200px 或其它非 1440px 作为最大宽度。
- 机会矩阵与业务动作清单继续使用所有方向混在一起的横向大表。
- 主题详情页或观点筛选状态只展示代表性评论而不展示全量相关评论。
